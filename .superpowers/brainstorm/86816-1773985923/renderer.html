Graph Renderer

1,006 nodes is the threshold where SVG starts struggling. Which rendering approach?

A

D3 force + Canvas 2D

D3 runs the physics simulation, draws to an HTML5 Canvas each frame. Smooth at 1k+ nodes, handles zoom/pan natively. Click detection requires manual hit-testing.

Pros

  • 60fps at 1k+ nodes
  • Familiar D3 API
  • Simple stack

Cons

  • Manual hit-testing for clicks
  • No DOM per-node (labels need redraw)
B

Sigma.js + WebGL

Purpose-built graph library with WebGL renderer. Handles 10k–100k nodes. Native click/hover, built-in zoom, label rendering. Used by Obsidian's graph view internally.

Pros

  • WebGL = GPU accelerated
  • Built-in node interaction
  • Obsidian-proven at this scale

Cons

  • Extra dependency
  • Less D3 flexibility
C

D3 force + SVG

Classic approach: each node is a real DOM element. Easy tooltips, hover states, CSS styling. Gets choppy above ~500 nodes but fine if we cluster/filter first.

Pros

  • Simple, zero extra deps
  • CSS hover/selection trivial
  • Labels always sharp

Cons

  • ~500 node limit for 60fps
  • Requires smart culling/LOD
Context: Your vault has 1,006 notes. With a 0.7 similarity threshold the edge count will likely be 5k–30k edges — comfortably in D3+Canvas territory. At 0.5 threshold it could balloon to 100k+ edges, where WebGL starts to matter.
